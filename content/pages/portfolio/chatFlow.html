ChatFlow 💬

A full-stack, cross-platform real-time messaging app built with React Native, Node.js, Firebase, and WebSockets.

Overview

ChatFlow is a production-grade messaging application that runs on iOS, Android, and Web from a single shared codebase. It delivers real-time messaging via Socket.IO WebSockets, supports media sharing up to 25MB via Cloudinary, and handles authentication through Firebase Auth with both Email/Password and Google OAuth on all platforms.

Every user gets a globally unique, permanent username enforced atomically at the database level — no auto-numbering fallbacks, no race conditions. The app includes real-time presence indicators, typing indicators with multi-user support, read receipts, an in-app image lightbox, emoji picker, and full dark mode throughout.

Key Features

⚡
Real-time Messaging
Bi-directional WebSocket communication via Socket.IO. Typing indicators with multi-user support, read receipts, and instant message delete synced across all connected clients in real time.
📸
Media Sharing & Lightbox
Send images, videos, and files up to 25MB via Cloudinary. Tap any image to expand it full-screen in an in-app lightbox, with save-to-Photos support on mobile.

One codebase powers all three platforms. The mobile app is built with React Native (Expo) and the web client with React + Vite, sharing the same backend, socket layer, and Firebase project.
👤
Unique Usernames & User Search
Usernames are globally unique, chosen at sign-up, and locked permanently — enforced via a dedicated Firestore collection. Find any user by email or @username with live search.

How It's Built

The backend is a Node.js + Express server running Socket.IO for all real-time events — messaging, typing indicators, presence, and conversation sync. It's deployed to Railway and authenticated via Firebase ID tokens verified server-side using the Firebase Admin SDK.

The React + Vite web client and React Native (Expo) mobile app share the same Firebase project and backend. Both use context providers for auth and socket state so any component can subscribe to real-time updates without prop drilling.

Media uploads go directly from the client to Cloudinary via the backend upload route. The CI/CD pipeline runs on GitHub Actions — push to main and the web app deploys to Firebase Hosting and the server deploys to Railway automatically.

Challenges & Learnings

The hardest problem was enforcing globally unique usernames without race conditions. The solution was a dedicated usernames Firestore collection where the document ID is the username itself — Firestore's atomic writes make it impossible for two users to claim the same username simultaneously, and a 409 error is returned instead of silently generating a fallback.

Getting Google OAuth to work consistently across web and both mobile platforms (iOS and Android) with the same Firebase project required separate client IDs per platform and careful handling of the auth redirect flow in Expo — especially on physical devices where localhost doesn't resolve and the local IP must be used instead.
